## Architecture Overview

### Server Role: Device-Platform
This server implements the **Device-Platform** role in the Farm IoT Interoperability Standard:
- **Receives**: Sensor data from devices via MQTT (existing)
- **Stores**: Time-series data on disk in `data/{org}/{project}/{device-id}/`
- **Serves**: Historical data to Farm-Platforms via standardized API (NEW)
- **Sends Commands**: To devices via MQTT through logger's MQTT client (NEW)
- **Manages**: Device schemas (dynamically generated by logger)

### Logger Integration
The logger (frugal-iot-logger) already:
- Connects to MQTT brokers
- Knows which modules are associated with each device
- Listens to all device topics

**New responsibilities for logger**:
- Generate device schemas dynamically (API call to be defined)
- Provide MQTT client interface for command delivery (API call to be defined)

## Implementation Phases

### Phase 1: Foundation & Infrastructure
**Goal**: Set up testing framework, SenML support, and API middleware

#### 1.1 Add Test Framework
- Install Jest or Mocha
- Create `test/` directory structure
- Write tests for existing functionality:
  - Authentication/session flow
  - OTA endpoints
  - Static file serving
  - Config.json generation

#### 1.2 Add SenML NPM Package
- Research and install SenML package (e.g., `senml-lite` or similar)
- Create SenML utility module: `lib/senml-utils.js`
  - `validateSenMLInput(packet)` - Basic validation for received packets
  - `toSenML(data, deviceId, baseTime)` - Convert internal format to SenML
  - `parseSenML(packet)` - Extract data from SenML format

#### 1.3 API Foundation
- Create `lib/api-errors.js` - Standardized error responses (per API.md Section 3.5)
  - Error codes: `invalid_request`, `not_authenticated`, `not_allowed`, `device_not_found`, `user_not_found`, `already_exists`, `invalid_value`, `field_read_only`, `device_unavailable`, `server_error`
  - Response format: `{ error: "code", message: "human-readable" }`
- Create Express middleware:
  - `apiErrorHandler` - Catch and format API errors
  - `platformAuthMiddleware` - Token-based auth for Farm-Platform requests (not session-based)

#### 1.4 Platform Registration Documentation
- Create `config.d/platforms/README.md`
- Define manual registration process:
  - How to create platform config files
  - Required fields: base URL, tokens, cookie names
  - Example config: `config.d/platforms/litefarm.yaml`

### Phase 2: Implement Farm-Platform Request Endpoints (API.md Section 6)
**Goal**: Allow Farm-Platforms to request data and interact with devices

#### 2.1 GET /data - Request Historical Data
**Endpoint**: `GET /data?device={device-id}&from={timestamp}&to={timestamp}`
- Parse device identifier (format: `org/project/device-id`)
- Load data from `data/{org}/{project}/{device-id}/` directory
- Filter by timestamp range
- Convert to SenML format using utility
- Return SenML packet or empty array if no data
- Tests: timestamp parsing, file loading, SenML conversion, edge cases

#### 2.2 POST /users/register - Register Farm-Platform User
**Endpoint**: `POST /users/register`
- Body: `{ "user-id": "farm-platform-user-id", "credentials": {...} }`
- Create user in database if not exists
- Return: `{ "status": "registered", "device-platform-user-id": "id" }`
- Error: 409 if already exists
- Tests: new user, duplicate user, database errors

#### 2.3 POST /devices/register - Register Device to User
**Endpoint**: `POST /devices/register`
- Body: `{ "user-id": "...", "farm-platform-device-id": "...", "metadata": {...} }`
- Verify user exists, device exists
- Create device-user mapping in database
- Store farm-platform device ID
- Return: `{ "status": "registered", "device-platform-device-id": "id" }`
- Errors: 404 user/device not found, 409 already registered
- Tests: valid registration, missing user/device, duplicates

#### 2.4 POST /devices/command - Send Command to Device
**Endpoint**: `POST /devices/command`
- Body: `{ "device-id": "...", "command": "module/field", "parameters": { "value": ... } }`
- Validate device exists
- Validate command field exists in device schema (call to logger)
- Validate value against schema type/min/max/rw
- Send command via MQTT through logger's MQTT client (new API call)
- Return: `{ "status": "accepted|rejected", "reason": "..." }`
- Errors: 404 device not found, 422 invalid value/field read-only, 503 device unavailable
- Tests: valid command, invalid field, wrong type, device offline

#### 2.5 GET /devices/schema?device={device-id} - Get Device Schema
**Endpoint**: `GET /devices/schema?device={device-id}`
- Parse device identifier
- Call logger API to get dynamic schema
- Return JSON conforming to Annex A structure
- Example response:
  ```json
  {
    "device-platform-device-id": "dev/org/esp32-123456/",
    "farm-platform-device-id": "farm-device-1",
    "modules": {
      "sht": {
        "fields": [
          {"field": "temperature", "type": "float", "units": "Cel", "rw": "r"}
        ]
      }
    }
  }
  ```
- Tests: valid device, device not found, schema generation

- Skeleton implementation

### Phase 4: Data Route Refactoring
**Goal**: Ensure GET /data serves both static files and SenML-formatted sensor data

#### 4.1 GET /data Route Analysis
Current behavior: Serves static files from `data/{org}/{project}/{device-id}/` directory

New behavior: 
- If request has query params `?from=` and `?to=` → return SenML-formatted data
- If request is path-based → return static files (backward compatible)

Implementation:
- Check for query parameters first
- If present, use Phase 2.1 logic
- If absent, serve static files as before

Tests: backward compatibility with existing file serving, new SenML response

## Database Schema Changes

### New Tables

#### farm_platforms
```sql
CREATE TABLE farm_platforms (
  id INTEGER PRIMARY KEY,
  name TEXT UNIQUE NOT NULL,
  base_url TEXT NOT NULL,
  auth_token TEXT NOT NULL,
  cookie_name TEXT NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### users_farm_platform
```sql
CREATE TABLE users_farm_platform (
  id INTEGER PRIMARY KEY,
  user_id TEXT NOT NULL,
  farm_platform_id TEXT NOT NULL,
  device_platform_user_id TEXT NOT NULL,
  UNIQUE(user_id, farm_platform_id),
  FOREIGN KEY(farm_platform_id) REFERENCES farm_platforms(name)
);
```

#### device_farm_mappings
```sql
CREATE TABLE device_farm_mappings (
  id INTEGER PRIMARY KEY,
  user_id TEXT NOT NULL,
  device_id TEXT NOT NULL,
  farm_platform_device_id TEXT NOT NULL,
  farm_platform_id TEXT NOT NULL,
  UNIQUE(user_id, device_id, farm_platform_id),
  FOREIGN KEY(farm_platform_id) REFERENCES farm_platforms(name)
);
```

#### data_push_queue
```sql
CREATE TABLE data_push_queue (
  id INTEGER PRIMARY KEY,
  farm_platform_id TEXT NOT NULL,
  senml_packet TEXT NOT NULL,
  retry_count INTEGER DEFAULT 0,
  last_retry DATETIME,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(farm_platform_id) REFERENCES farm_platforms(name)
);
```

## Backward Compatibility Analysis

### Existing Endpoints to Preserve
- `GET /` - Static UI serving ✓
- `GET /login` - Login form ✓
- `POST /login` - Authentication ✓
- `POST /register` - User registration ✓
- `GET /dashboard` - Dashboard ✓
- `GET /config.json` - Config (may need tweaks)
- `GET /data/{path}` - Static file serving (CAREFUL - may conflict with new /data route)
- `GET /ota_update/:org/:project/:node/:attribs` - OTA download ✓
- `POST /ota_update` - OTA upload ✓
- `GET /private/*` - Private files ✓

### Potential Issues
1. **GET /data route conflict**: 
   - Current: Serves static files from `data/` directory
   - New: Also serves SenML-formatted data with query params
   - Solution: Check for `?device=`, `?from=`, `?to=` query params first; if absent, fall through to static file serving
